# Foreman ‚Äî Architecture Reference

This document is the **canonical reference for codebase conventions**. Before adding any new feature, read this file. Its purpose is to ensure new resources are built consistently with existing patterns.

---

## Layer Order

Every resource follows this top-down layering:

```
migrations/versions/   ‚Üê SQL DDL (Alembic, raw SQL ‚Äî no ORM)
foreman/models/        ‚Üê Internal dataclass mirroring the DB row
foreman/schemas/       ‚Üê Pydantic schemas for HTTP I/O
foreman/repositories/  ‚Üê async CRUD functions (one module per resource)
foreman/api/v1/endpoints/  ‚Üê FastAPI route handlers
foreman/main.py        ‚Üê Router registration
tests/                 ‚Üê Pytest unit tests (mocked CRUD, no real DB)
```

---

## Database Conventions

| Convention | Detail |
|---|---|
| **Primary keys** | `UUID`, always generated by Postgres via `gen_random_uuid()` |
| **Created timestamp** | `created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP` |
| **Updated timestamp** | `updated_at TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP` (nullable) |
| **SQL style** | Raw SQL only. Use the `sql()` helper from `foreman.db` ‚Äî **no ORM** |
| **Migrations** | Alembic, sequential numeric prefix: `0001_`, `0002_`, etc. |
| **JSON columns** | Use `JSONB` |

---

## Delete Strategy

| Resource type | Strategy |
|---|---|
| `users` | **Soft delete** ‚Äî set `is_deleted=TRUE, is_active=FALSE` |
| Child resources (projects, generations, ‚Ä¶) | **Hard delete** with `ON DELETE CASCADE` FK |

The `CASCADE` rule means you only need to delete the parent row; child rows are removed by the DB automatically.

---

## Authentication & Authorization

- Auth is resolved via the `X-User-ID` HTTP header, which a gateway sets after verifying the token.
- The `get_current_user` dependency in `foreman/api/deps.py` reads this header, looks up the user, and raises `401` if anything is wrong.
- **Every repository query for an owned resource MUST include `AND user_id=$n`** in the `WHERE` clause to prevent horizontal privilege escalation (user A reading user B's data).

## Model Layer (`foreman/models/`)

Plain `@dataclass` ‚Äî no Pydantic, no ORM. Fields map 1-to-1 with the DB columns.

```python
@dataclass
class Project:
    id: uuid.UUID
    user_id: uuid.UUID
    name: str
    original_image_url: Optional[str]
    room_analysis: Optional[dict]
    created_at: datetime
    updated_at: Optional[datetime]
```

Construct from an asyncpg `Record` via `Model(**dict(record))`.

---

## Schema Layer (`foreman/schemas/`)

Three Pydantic schemas per resource:

| Schema | Purpose | Notes |
|---|---|---|
| `ResourceCreate` | POST body | Only required + optional create fields |
| `ResourceUpdate` | PATCH body | All fields optional; `model_config = ConfigDict(extra="forbid")` |
| `ResourceRead` | Response body | All fields; `model_config = ConfigDict(from_attributes=True)` |

---

## Repository Layer (`foreman/repositories/`)

- One **module** of free async functions per resource (no classes).
- Name pattern: `postgres_<resource>_repository.py`
- Import as `crud` in endpoints: `from foreman.repositories import postgres_projects_repository as crud`
- Use the `sql()` helper for all statements.
- For partial updates, build dynamic `SET` clauses using an `ALLOWED_UPDATE_FIELDS` frozenset to guard against column injection ‚Äî same pattern as `postgres_users_repository.py`.
- Always `RETURNING *` on mutations so you can reconstruct the model without a second query.

---

## Endpoint Layer (`foreman/api/v1/endpoints/`)

- One `APIRouter()` per resource file.
- Always inject both `get_current_user` and `get_db` via `Depends`.
- Return `404` when a record is not found (or not owned by the caller ‚Äî do **not** distinguish between "not found" and "not yours" to avoid leaking existence).
- Wrap mutations in `try/except Exception` and, as a last resort, log the exception with a full stack trace and raise `HTTPException(status_code=500, detail="Internal server error")` (never expose raw exception messages like `str(e)` to clients).
- Register the router in `main.py` with prefix `/v1/<resource>` and a matching tag.

---

## Pagination

List endpoints accept two query parameters:

| Param | Default | Max | Description |
|---|---|---|---|
| `limit` | `20` | `100` | Maximum records returned |
| `offset` | `0` | ‚Äî | Number of records to skip |

Enforce bounds with `Query(20, ge=1, le=100)` and `Query(0, ge=0)`.

## Test Layer (`tests/`)

- **No real database**. Use in-memory `dict` stores and `monkeypatch` to replace CRUD functions.
- Override `get_db` to return `None` (mocked CRUD never uses it).
- Override `get_current_user` with a local version that reads `X-User-ID` from an in-memory users dict.
- Name patch targets as the fully qualified attribute seen by the endpoint module, e.g. `"foreman.api.v1.endpoints.projects.crud.create_project"`.
- Reset all stores and `app.dependency_overrides` in `yield` teardown.

### Mandatory test cases for every resource

1. List ‚Äî empty, then populated
2. Create ‚Äî 201, correct fields returned
3. Get by ID ‚Äî found, not found (404)
4. Update ‚Äî partial update works
5. Delete ‚Äî 204, subsequent GET returns 404
6. Ownership ‚Äî another user's ID gets 404 on get/patch/delete
7. Unauthenticated ‚Äî missing header returns 401
8. Validation ‚Äî missing required fields return 422

### Test Coverage

- **Aim for 100% code coverage** on new features.
- Run `pytest --cov=foreman --cov-report=term-missing` to check coverage.
- New modules require unit tests in `tests/`.
- Integration with existing endpoints requires added test cases to existing test files.

## Environment Variables

All new environment variables must be documented in `.env.foreman.example`.

| Convention | Detail |
|---|---|
| **Documentation** | Every new env var must have a comment describing its purpose |
| **Defaults** | Use sensible defaults in code; mark optional vars with `# optional` |
| **Secrets** | Never commit actual secrets; use placeholder values like `your_access_key` |

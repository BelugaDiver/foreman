# Foreman Constitution

## Core Principles

### I. Layered Architecture (NON-NEGOTIABLE)

Every resource in the system MUST follow this strict top-down layer order with
no cross-layer skipping:

```
migrations/versions/       ← Raw SQL DDL only (Alembic, sequential prefix 0001_…)
foreman/models/            ← Plain @dataclass mirroring DB columns 1-to-1
foreman/schemas/           ← Pydantic v2 schemas for HTTP I/O (three per resource)
foreman/repositories/      ← Async CRUD functions (one module per resource, no classes)
foreman/api/v1/endpoints/  ← FastAPI route handlers (one APIRouter per file)
foreman/main.py            ← Router registration only
```

- The worker (`worker/`) is a separate deployable process that shares
  `foreman/` models, repositories, and database utilities — it MUST NOT
  duplicate any of those layers.
- Adding a new resource means adding a file (or migration) at every layer in
  order. Skipping a layer or merging two layers into one is a violation.
- `instructions/ARCHITECTURE.md` is the canonical reference and MUST be kept
  in sync with this constitution whenever a layer rule changes.

**Rationale**: Enforces predictable navigability, prevents business logic from
leaking into route handlers, and makes isolated unit testing achievable without
a real database.

### II. Raw SQL Only — No ORM (NON-NEGOTIABLE)

All database access MUST use the `sql()` helper from `foreman.db` with
positional parameters (`$1`, `$2`, …).

- No SQLAlchemy ORM, no query-builder DSLs. Raw SQL is the only dialect.
- Every mutation MUST include `RETURNING *` so the result can be reconstructed
  from a single round-trip without a follow-up SELECT.
- Partial UPDATE statements MUST filter candidate columns through an
  `ALLOWED_UPDATE_FIELDS: frozenset` guard to prevent column-injection.
- Migrations are Alembic-managed raw SQL files with the naming pattern
  `<NNNN>_<description>.py`, starting at `0000`.
- Primary keys MUST be UUID generated by `gen_random_uuid()` in Postgres.
- Timestamps MUST be `TIMESTAMPTZ`; `created_at NOT NULL DEFAULT CURRENT_TIMESTAMP`,
  `updated_at` nullable.
- JSON columns MUST use `JSONB`.
- Soft-delete applies to `users` only (`is_deleted=TRUE, is_active=FALSE`).
  All child resources use hard-delete with `ON DELETE CASCADE` foreign keys.

**Rationale**: Keeps SQL visible, reviewable, and free of hidden query magic.
Parameterized queries are the primary SQL-injection defense.

### III. Async-First Throughout

All I/O — database queries, HTTP calls, queue operations, file uploads, and AI
model invocations — MUST use `async`/`await`.

- Blocking calls (boto3, subprocess) MUST be wrapped with `asyncio.to_thread()`
  to avoid stalling the event loop.
- pytest MUST run with `asyncio_mode = "auto"` (already configured in
  `pyproject.toml`); test functions that need async MUST be declared `async`.
- Synchronous helper functions (pure computation, formatting) are acceptable
  and SHOULD remain synchronous.

**Rationale**: FastAPI and asyncpg are async-native; mixing blocking and
non-blocking code causes latency cliffs and exhausts the thread pool.

### IV. Event-Driven Processing via Decoupled Queue

Image generation requests MUST be dispatched through the queue layer; the
Foreman API MUST NOT call AI providers directly.

- The API publishes a `QueueMessage` (via `QueueProtocol`) and returns `202
  Accepted`.
- The Worker consumes messages from the queue, calls the AI provider, uploads
  the result to object storage, and updates the generation status via the
  repository layer.
- The queue and storage backends are selected at runtime via environment
  variables (`QUEUE_PROVIDER`, `STORAGE_PROVIDER`); the rest of the codebase
  always interacts through the abstract `QueueProtocol` and `StorageProtocol`
  interfaces.
- The Worker MUST honour a configurable `max_retries` count and delete
  malformed or persistently failing messages to avoid poison-pill loops.
- Graceful shutdown MUST drain in-flight jobs within a configurable timeout
  before terminating worker processes.

**Rationale**: Decoupling allows independent scaling of the API and Worker,
ensures expensive AI calls are retried reliably, and prevents long-running
jobs from blocking HTTP request handling.

### V. Protocol-Abstracted External Dependencies

Every external integration (queue, object storage, AI provider) MUST be
accessed through an `ABC`-based protocol class and injected via factory
functions or FastAPI `Depends`.

- `QueueProtocol` — `publish()` / `close()` — current implementation: AWS SQS.
- `StorageProtocol` — `create_upload_url()` / `get_download_url()` / `delete()`
  — current implementation: Cloudflare R2.
- AI providers are selected via `AI_PROVIDER` env var and accessed through the
  factory in `worker/providers/`.
- No endpoint or processor MUST import a concrete provider class directly; it
  MUST always work through the protocol or a factory function.

**Rationale**: Swapping providers (e.g., SQS → Google Pub/Sub, R2 → S3) MUST
require changing only the factory and the concrete class, not business logic.

### VI. Test-Layered Coverage (NON-NEGOTIABLE)

Two distinct test layers are REQUIRED; both MUST pass in CI before a PR merges:

| Layer | Location | Database | Goal |
|---|---|---|---|
| Unit | `tests/foreman/`, `tests/worker/` | None — in-memory `dict` stores + `monkeypatch` | Fast, isolated, 100% path coverage |
| Integration | `tests/foreman/integration/` | Real PostgreSQL via testcontainers | Verify SQL, constraints, transactions |

**Unit test rules** (enforced):
- MUST NOT touch a real database; `get_db` is overridden to return `None`.
- MUST patch CRUD at the fully-qualified module path seen by the endpoint,
  e.g. `"foreman.api.v1.endpoints.projects.crud.create_project"`.
- Every endpoint test file MUST have an `autouse=True` fixture called
  `mock_dependencies` that seeds stores, registers overrides, and tears down
  completely in `yield` teardown.
- Every test function MUST have a one-line docstring following the pattern:
  `[<METHOD>] <path> [+ condition] should <expected result>.`
- Test bodies MUST follow the AAA structure with `# Arrange`, `# Act`, and
  `# Assert` comments.
- Every new resource MUST cover the 13 mandatory scenarios listed in
  `instructions/ARCHITECTURE.md`.

**Integration test rules**:
- MUST truncate all tables before AND after each test for isolation.
- MUST use `testcontainers` to spin up a real Postgres instance; no
  assumptions about an external DB.
- MUST run Alembic migrations against the container before the suite begins.

**Coverage gate** (CI-enforced): 85% minimum across `foreman/` and `worker/`.
New modules SHOULD target 100% on business-critical paths.

**Rationale**: Fast unit tests catch regressions in seconds; integration tests
validate SQL correctness against real Postgres constraints and FK rules that
mocks cannot exercise.

### VII. Observability by Default

Every component MUST emit telemetry without requiring opt-in per feature:

- **Distributed tracing**: OpenTelemetry spans MUST wrap every significant
  operation (HTTP handlers via `FastAPIInstrumentor`, queue polling, AI
  generation, storage uploads). Spans MUST record relevant attributes
  (`generation_id`, `project_id`, status codes) but MUST NOT record PII or
  prompt content (use `prompt_length` instead of `prompt`).
- **Structured logging**: Production MUST use JSON-formatted logs
  (`LOG_FORMAT=json`). All log statements MUST use `get_logger(__name__)` —
  never `print()`. Log levels MUST be appropriate (`DEBUG` for internals,
  `INFO` for lifecycle events, `WARNING` for recoverable issues, `ERROR`/
  `EXCEPTION` for failures).
- **Correlation IDs**: Every HTTP request MUST receive a correlation ID
  (from `X-Request-ID` header or auto-generated UUID) propagated through
  `ContextVar` and echoed in the `X-Correlation-ID` response header and every
  log record emitted during that request.
- **Audit log**: All user-initiated mutations (create, update, delete) on
  users, projects, generations, and images MUST call `log_audit()` via the
  `foreman.audit` module with the correct `AuditEvent` enum value.
- **Health and readiness**: Both the API (`/health`) and Worker (`/health`,
  `/ready`) MUST expose liveness and readiness endpoints; the Worker readiness
  endpoint MUST verify DB connectivity and consumer state.

**Rationale**: Observability is not optional in an event-driven system where
failures may surface asynchronously in the worker, long after the HTTP
response was returned.

### VIII. Security at Every Layer (NON-NEGOTIABLE)

- **Authorization — ownership enforcement**: Every repository query for
  user-owned resources MUST include `AND user_id=$n` in the `WHERE` clause.
  Endpoints MUST return `404` (not `403`) when a resource is not found or not
  owned by the caller — do NOT leak existence information.
- **Authentication**: Auth is resolved via the `X-User-ID` header set by the
  upstream gateway. The `get_current_user` FastAPI dependency validates the
  UUID, confirms the user exists and is active, and raises `401` otherwise.
- **SQL injection prevention**: All database access MUST use the parameterized
  `sql()` helper. String interpolation into SQL queries is forbidden.
- **SSRF prevention**: Any code that fetches remote URLs (e.g., worker image
  downloader) MUST validate resolved IP addresses against a blocklist of
  private/loopback/link-local ranges, check `Content-Length` before
  downloading, and enforce a hard maximum file size.
- **Error exposure**: Raw exception messages (`str(exc)`) MUST NEVER be
  returned to API clients. Endpoint handlers MUST catch unexpected exceptions
  and return `HTTPException(status_code=500, detail="Internal server error")`.
  Full stack traces MUST be logged server-side using `logger.exception()`.
- **Secrets**: Secrets (API keys, DB credentials, cloud credentials) MUST
  never be committed to the repository. All secrets MUST be supplied via
  environment variables. Every env var MUST be documented with a comment in
  `.env.foreman.example` using a placeholder value (e.g., `your_access_key`).
- **CORS**: `CORS_ORIGINS` MUST be restricted to known domains in production;
  the wildcard `*` default is only acceptable in local development.

**Rationale**: Horizontal privilege escalation (user A reading user B's data)
is the primary application-level security risk given the multi-tenant model;
ownership enforcement in every query is the primary mitigation.

## Technology Stack & Tooling

### Runtime

| Concern | Technology | Notes |
|---|---|---|
| Language | Python 3.11+ | Minimum version; 3.14 used locally |
| Web framework | FastAPI | Async, OpenAPI auto-docs at `/docs` and `/redoc` |
| ASGI server | Uvicorn (standard) | Used directly in Docker; `--reload` for local dev |
| Database | PostgreSQL 16+ | asyncpg driver; no ORM |
| Migrations | Alembic + raw SQL | `alembic upgrade heads` before first run |
| Message queue | AWS SQS | Abstracted; `QUEUE_PROVIDER=none` disables |
| Object storage | Cloudflare R2 | S3-compatible; abstracted via `StorageProtocol` |
| AI provider | Google Vertex (Gemini) | Abstracted; `AI_PROVIDER=vertex` |
| Tracing | OpenTelemetry + Jaeger | OTLP HTTP export; Jaeger UI on `:16686` |

### Development Tooling

| Tool | Purpose | Invocation |
|---|---|---|
| `ruff check .` | Lint (E, F, I, N, W rules) | Must pass — no warnings allowed |
| `ruff format .` | Format (100-char line length) | Must pass before commit |
| `pytest` | Test runner | Unit: `pytest tests/ --ignore=tests/foreman/integration` |
| `pytest-asyncio` | Async test support | `asyncio_mode = "auto"` |
| `testcontainers` | Integration DB | `pytest tests/foreman/integration/` |
| `moto` | AWS mock (SQS) | Used in unit tests for queue layer |
| `httpx` | Test HTTP client | Used in integration tests |
| `alembic` | Schema migration | `alembic upgrade heads` |
| `docker-compose` | Local full stack | Postgres + Jaeger + API + Worker |

### Environment Configuration

- All environment variables MUST be listed in `.env.foreman.example` with
  descriptive comments and safe placeholder values.
- Local overrides go in `.env.foreman.local` (gitignored).
- Docker Compose loads from `.env.foreman`.
- `DEV_MODE=true` seeds a canonical dev test user on startup; MUST NOT be
  enabled in production.

## Code Style & Naming Conventions

### Naming

| Context | Convention | Example |
|---|---|---|
| Variables, functions, modules | `snake_case` | `get_current_user`, `user_id` |
| Classes | `PascalCase` | `Database`, `GenerationRead` |
| Constants | `UPPER_CASE` | `ALLOWED_UPDATE_FIELDS`, `MAX_DOWNLOAD_BYTES` |
| Private methods/attributes | `_leading_underscore` | `_pool`, `_get_client()` |
| Test files | `test_<resource>.py` | `test_projects.py` |
| Repository modules | `postgres_<resource>_repository.py` | `postgres_users_repository.py` |

### Code Organization Rules

- Functions SHOULD be under 50 lines; extract helpers when they grow beyond that.
- Circular imports are forbidden; use `__init__.py` sparingly to expose public APIs.
- Import order: stdlib → third-party → local, each group separated by a blank
  line (enforced by `ruff`'s `I` rule).
- Type hints are REQUIRED on all function signatures and class attributes.
- Docstrings are REQUIRED on all public functions, classes, and modules.
  Use Google-style or NumPy-style consistently.
- Use f-strings for string formatting; `%`-style or `.format()` are discouraged.
- Prefer `pathlib` over `os.path` for file system operations.

### Schema Conventions

- Three Pydantic v2 schemas per resource: `<Resource>Create`, `<Resource>Update`,
  `<Resource>Read`.
- Update schemas MUST use `ConfigDict(extra="forbid")` to reject unknown fields.
- Read schemas MUST use `ConfigDict(from_attributes=True)` for ORM-style init.
- Field validators MUST be class methods decorated with `@field_validator`.

### Pagination Conventions

- List endpoints MUST accept `limit` (default 20, max 100) and `offset`
  (default 0) query parameters, enforced with `Query(20, ge=1, le=100)` and
  `Query(0, ge=0)`.

### Error Handling Conventions

- Use domain exceptions from `foreman/exceptions.py` (`ResourceNotFoundError`,
  `DuplicateResourceError`, `InvalidStateError`) inside the repository layer.
- Endpoint handlers translate domain exceptions to `HTTPException` with the
  appropriate HTTP status code.
- A bare `except Exception` catch MUST always be the last resort inside an
  endpoint; it MUST log the full traceback with `logger.exception()` and raise
  `HTTPException(500, "Internal server error")` — never `str(exc)`.
- Global exception handlers in `main.py` translate infrastructure exceptions
  (asyncpg connection failures, S3 `ClientError`) to `503` responses with
  generic messages.

## Development Workflow

### Git & CI

- The `tests.yml` GitHub Actions workflow MUST pass on every PR targeting
  `main` or `master`:
  1. Unit tests: `pytest tests/foreman/ --ignore=tests/foreman/integration`
  2. Worker unit tests: `pytest tests/worker/`
  3. Integration tests: `pytest tests/foreman/integration/`
  4. Coverage check: ≥85% across `foreman/` and `worker/` combined.
- Commits MUST be focused on a single change and reference issue numbers where
  applicable.
- Build artifacts, `__pycache__`, `.env.foreman` (with real secrets), and
  `htmlcov/` MUST NOT be committed (covered by `.gitignore`).

### Adding a New Resource (Checklist)

Follow this order without skipping steps:

1. Add Alembic migration: `migrations/versions/<NNNN>_create_<resource>_table.py`
2. Add dataclass model: `foreman/models/<resource>.py`
3. Add Pydantic schemas: `foreman/schemas/<resource>.py` (Create / Update / Read)
4. Add repository module: `foreman/repositories/postgres_<resource>_repository.py`
5. Add endpoint router: `foreman/api/v1/endpoints/<resource>.py`
6. Register router in `foreman/main.py` with prefix `/v1/<resource>` and tag
7. Add unit test file: `tests/foreman/test_<resource>.py` covering all 13 mandatory scenarios
8. Add integration tests: `tests/foreman/integration/test_<resource>.py`
9. Document new env vars (if any) in `.env.foreman.example`

### Constitution Check Gates (for plan.md)

Before implementing any feature, verify:

- [ ] Every new data entity follows the 6-layer order (Principle I).
- [ ] No SQLAlchemy or query-builder imports introduced (Principle II).
- [ ] No blocking I/O in async functions without `asyncio.to_thread()` (Principle III).
- [ ] Generation work dispatched via queue, not called synchronously (Principle IV).
- [ ] New external integrations implement and use the protocol ABC (Principle V).
- [ ] Unit tests use in-memory mocks; integration tests use testcontainers (Principle VI).
- [ ] New spans added for significant operations; no PII in span attributes (Principle VII).
- [ ] All owned-resource queries include `AND user_id=$n` (Principle VIII).
- [ ] No `str(exc)` in API responses; no secrets in code (Principle VIII).

## Governance

This constitution supersedes all prior undocumented conventions and any
conflicting guidance in other files. In case of conflict, this file prevails;
the conflicting guidance in the other file MUST be updated within the same PR
that causes the conflict.

**Amendment procedure**:
1. Create a PR with the proposed change to this file and all affected artifacts.
2. Increment `CONSTITUTION_VERSION` following SemVer:
   - MAJOR: removing or redefining a principle in a backward-incompatible way.
   - MINOR: adding a new principle or materially expanding a section.
   - PATCH: clarifications, wording fixes, non-semantic refinements.
3. Update `LAST_AMENDED_DATE` to the merge date.
4. Update `instructions/ARCHITECTURE.md` and `instructions/TESTING.md` in the
   same PR if the amendment changes their guidance.
5. Update `.specify/templates/plan-template.md` Constitution Check gates if
   principles are added or removed.

**Compliance**:
- All code reviews MUST verify compliance with Principles I, II, VI, and VIII
  as a minimum gate before approval.
- Deviations from any NON-NEGOTIABLE principle require explicit justification
  documented in `plan.md` under "Complexity Tracking" before implementation.
- For runtime development guidance, consult `instructions/ARCHITECTURE.md`
  (layer conventions) and `instructions/TESTING.md` (test conventions).

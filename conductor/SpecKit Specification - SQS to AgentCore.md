# SpecKit Specification: SQS to AgentCore Img2Img Translation Engine

## 1. System Overview & Objective
The system is an asynchronous, decoupled Image-to-Image (Img2Img) translation processing engine. A pre-existing Python worker running inside a Docker container long-polls an AWS SQS queue for incoming translation tasks. 

Instead of processing heavy image generation loops locally, the worker delegates the reasoning, session memory, and model execution to **Amazon Bedrock AgentCore Runtime**. To bypass expensive AWS VPC network egress costs, raw image binaries generated by the model are written directly to an Amazon S3 bucket from the AgentCore execution layer. The agent returns only a lightweight resource reference key back to the SQS worker to update the database and complete the lifecycle.

## 2. Component Architecture & Data Flow
1. **Message Ingestion**: The Python worker pulls an inbound job from SQS, extracts the metadata, and maps the unique tracking identifier to an AgentCore `sessionId`.
2. **AgentCore Invocation**: The worker routes the textual prompt and source image URI to the cloud-hosted AgentCore instance using a thread pool executor to maintain concurrency.
3. **Img2Img Generation**: The agent uses its Strands reasoning core to pull the source image, invoke the generative model (e.g., Nova Canvas or Stable Diffusion), and perform the translation.
4. **Zero-Egress Save**: The generated binary is flushed directly out of the AgentCore sandbox into a private S3 bucket, returning only a string reference schema to the waiting worker thread.
5. **State Consolidation**: The worker thread receives the S3 key, updates the centralized database state, and acknowledges (deletes) the original message from SQS.

## 3. Data Contracts & Schemas

### 3.1 SQS Inbound Message Body (`PayloadSchema`)
```json
{
  "job_id": "job_99823_prod",
  "session_id": "session_user_77162_alpha",
  "source_image_s3_uri": "s3://my-app-vault/inputs/raw_user_photo.jpg",
  "translation_prompt": "Transform this image into a 1980s synthwave aesthetic, neon purple hues"
}

```

### 3.2 AgentCore Execution Return Chunk (`ResponseSchema`)

```json
{
  "status": "COMPLETED",
  "job_id": "job_99823_prod",
  "session_id": "session_user_77162_alpha",
  "artifact": {
    "bucket": "my-app-vault",
    "s3_resource_id": "outputs/session_user_77162_alpha/trans_99823.png"
  }
}

```

## 4. Functional Requirements

### 4.1 SQS Consumer Modifications (The Broker)

* **API Integration:** The worker **MUST** initialize the `bedrock-agent-runtime` client using `boto3`.
* **Session Mapping:** The worker **MUST** pass the incoming `session_id` payload element directly into the `sessionId` parameter of the `invoke_agent` API call. This ensures that the agent sticky-loads historical session memory.
* **Non-Blocking Execution:** The worker **MUST** delegate each `invoke_agent` processing call to a `ThreadPoolExecutor` so that network waiting states do not halt or throttle the main SQS line-polling loop.

### 4.2 AgentCore Tool Matrix (The Worker Layer)

* **Model Selection:** The agent workspace **MUST** be bound to a reasoning model (e.g., Claude 3.5 Sonnet) and feature direct tool mapping to an architectural image pipeline engine (e.g., Bedrock Flows running Stable Diffusion or Nova Canvas).
* **Direct Storage Handshake:** The inner image tool **MUST** accept the source image URI, process the translation, write the generated binary stream directly to the target S3 output path, and format the final string output payload as clean, stringified JSON containing the target resource ID.

## 5. Non-Functional & Operational Constraints

* **Network Egress Limits:** Raw binary images (`.png`, `.jpg`) **MUST NOT** be passed through the network stream between AgentCore and the SQS worker container. Only the valid stringified JSON response block containing the S3 resource keys is permitted.
* **Visibility Timeout Mapping:** Because multi-turn image generation, translation, and S3 file flushing can take anywhere from 10 to 45 seconds, the SQS Queue visibility timeout **MUST** be configured to a minimum of 5 minutes to prevent race conditions or duplicate deliveries.
* **Security Bounds:** The local Docker container requires permissions *only* to read/delete from SQS and call `bedrock:InvokeAgent`. The actual S3 bucket write permissions (`s3:PutObject`) must reside solely within the IAM execution role attached to the cloud-hosted AgentCore runtime.

## 6. Target Project File Mappings

* `src/worker/consumer.py`: The updated SQS consumer loop utilizing thread-pool offloading to Bedrock Agent Runtime.
* `src/agent/app.py`: The AgentCore entrypoint managing the core tool definition.
* `src/agent/tools/img2img_tool.py`: The decoupled image generation function executing the direct S3 save sequence.

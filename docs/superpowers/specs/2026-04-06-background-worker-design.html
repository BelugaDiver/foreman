Background Worker Design - Generation Processing

Date: April 7, 2026

Status: Approved

1. Overview

The background worker consumes generation requests from SQS and processes them using a pluggable AI provider abstraction and a custom agent graph with iteration logic.

┌─────────────┐     SQS     ┌──────────────────┐    DB     ┌─────────────┐
│   API       │ ──────────→ │   Worker         │ ────────→ │  Postgres   │
│ (producer)  │   Queue     │ (consumer)       │           │  Database   │
└─────────────┘             └────────┬─────────┘           └─────────────┘
                                     │
                                     ↓
                            ┌──────────────────┐    Upload   ┌───────────┐
                            │  AI Provider     │ ──────────→ │    R2     │
                            │  Abstraction     │             │  Storage  │
                            └──────────────────┘             └───────────┘
                                     ↑
                                     │
                            ┌────────┴─────────┐
                            │ Your Agent Graph │  ← pluggable
                            │ (iteration loop) │
                            └──────────────────┘

2. Architecture Components

2.1 Worker Entry Point

New file: foreman/worker/main.py

  • Long-lived process with graceful shutdown handling
  • Loads configuration from environment
  • Creates SQS consumer, database connection, storage client
  • Runs worker loop that processes messages one at a time

2.2 Message Consumer

New file: foreman/worker/consumer.py

  • Long-polls SQS queue for messages
  • Parses message body into GenerationJob
  • Handles message deletion on success, visibility timeout on failure

2.3 Job Processor

New file: foreman/worker/processor.py

  • Coordinates the generation pipeline
  • Updates generation status in DB at each stage
  • Handles retry logic for transient errors
  • Uploads result to R2 on success

2.4 AI Provider Abstraction

New file: foreman/worker/providers/base.py and foreman/worker/providers/

  • Base protocol: generate(prompt, input_image_url) -> ImageResult
  • Pluggable providers: OpenAI DALL-E, Replicate, custom
  • Factory to instantiate provider based on config

2.5 Agent Graph Integration

New file: foreman/worker/agent.py

  • Wraps your custom agent graph
  • Interface: run(input_image_url, prompt, style_id) -> OutputResult
  • Runs iteration loop: analyze → enhance → generate → evaluate → repeat

2.6 Database Repository

Reuse: foreman/repositories/postgres_generations_repository.py

  • Update generation status: pending → processing → completed/failed
  • Set output_image_url, error_message, processing_time_ms

3. Message Format

Current SQS message (unchanged):

{
  "generation_id": "uuid",
  "project_id": "uuid", 
  "prompt": "make it modern",
  "style_id": "uuid-or-null",
  "input_image_url": "https://...",
  "created_at": "2026-04-06T12:00:00Z"
}

4. Processing Pipeline

4.1 Status Flow

Status Description Transitions to
pending Created, not yet processed processing (worker starts)
processing Worker is processing completed, failed, cancelled
completed Successfully generated — (terminal)
failed Generation failed — (client can retry)
cancelled User cancelled — (terminal)

4.2 Pipeline Steps

  1. Receive message — Parse generation_id, load generation from DB
  2. Verify status — Skip if already processing/completed/cancelled
  3. Update status to processing — Set start time
  4. Download input image — Get from input_image_url
  5. Run agent graph — Your custom pipeline with iteration loop:
    • Analyze input image
    • Enhance prompt (LLM call)
    • Generate image (AI provider)
    • Evaluate quality
    • Iterate if needed
  6. Upload output — Upload generated image to R2, get public URL
  7. Update DB — Set status=completed, output_image_url, processing_time_ms
  8. Delete message — Confirm successful processing

5. Error Handling & Retry

5.1 Error Categories

Category Examples Action
Transient API timeout, rate limit, network error Auto-retry (up to 3 attempts)
Business Logic Invalid prompt, image format error Fail immediately, set error_message
Non-Recoverable Generation cancelled by user Skip, mark as cancelled

5.2 Retry Logic

  • Max 3 retries for transient errors
  • Exponential backoff between retries (2s, 4s, 8s)
  • After max retries, update status to failed with error_message

6. Configuration

Environment Variable Description Default
WORKER_CONCURRENCY Number of concurrent workers 1
WORKER_MAX_RETRIES Max retries for transient errors 3
WORKER_POLL_INTERVAL SQS long-poll interval (seconds) 10
WORKER_VISIBILITY_TIMEOUT SQS visibility timeout (seconds) 300

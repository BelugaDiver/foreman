Worker Module Wiring: Current State and Bedrock Path

This diagram maps how the worker currently processes jobs and where Google Vertex is coupled into runtime configuration. It also marks the exact seams where Amazon Bedrock can be introduced behind the same provider contract.

Current Provider Selection

Configured via AI_PROVIDER, but factory currently supports only vertex.

Runtime Coupling

worker/main.py passes Google-specific kwargs regardless of provider type.

Main Extension Goal

Add bedrock provider implementation and provider-specific config branches in factory wiring.

Current Wiring (As Implemented)

Entry worker/main.py :: main()

Loads config, DB, telemetry, storage, and creates provider instance before starting SQS consumer.

get_worker_config() -> get_provider(...)
Factory worker/providers/__init__.py :: get_provider

Only branch: if provider_type == "vertex" returns GeminiProvider.

else: ValueError("Unknown provider")
Implementation worker/providers/vertex.py :: GeminiProvider

Uses Google GenAI Vertex client for prompt enhancement and image generation.

genai.Client(vertexai=True, ...)
Usage worker/processor.py :: JobProcessor._run_agent

Calls provider via a generic method shape:

await ai_provider.generate(...)
main -> factory -> provider implementation -> processor.generate()
  • Good seam already exists: JobProcessor depends on generate(...), not a concrete Vertex class.
  • Coupling point: provider initialization kwargs in worker/main.py are Google-specific today.
  • Coupling point: worker/config.py currently contains Google model/project fields only for AI settings.

Target Wiring (Provider-Configurable for Bedrock)

Config worker/config.py

Add Bedrock config fields and validation by provider type.

AI_PROVIDER=vertex|bedrock
Factory Split worker/providers/__init__.py

Create provider-specific constructor branches with explicit kwargs per provider.

if vertex: GeminiProvider(...)
if bedrock: BedrockProvider(...)
New Provider worker/providers/bedrock.py

Implement same generate(...) interface to keep processor unchanged.

class BedrockProvider: async def generate(...)
Stable worker/processor.py

No structural change needed if provider contract is preserved.

JobProcessor remains provider-agnostic
main/config -> factory branch -> vertex or bedrock -> same processor call
  • Introduce an explicit provider protocol (optional but recommended): generate(prompt, input_image_url, style_id, enhance_prompt).
  • Keep all provider-specific SDK setup inside provider modules, not in worker/main.py.
  • Add tests mirroring existing provider tests: factory selection, config validation, and processor compatibility.
